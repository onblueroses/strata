# Frontier Dispatch Instructions (verbatim dispatch text)

Two codex dispatches the spec workflow makes. Scope both to ONE frontier: the single live move, planned on contact. The session owns the North Star, Boundaries, and Territory; codex supplies horsepower inside that frame.

## Frontier planner (optional: heavy-architecture frontiers)

Send verbatim to the strong lane through `bin/strong`, prepending the task description, the spec's Goal / Success means / Stop when, Boundaries, Decisions journal, Trail entries, current Territory sketch, and applicable CLAUDE.md rules.

> Plan the single next frontier for this spec: one move of ~30-90 min toward the stated Goal, informed by the Trail. Return only the frontier content and leave file writing to the caller. Write: the frontier heading (`### Frontier N: [name] (~NN min)`); a `Needs:` line listing artifacts the move builds on (each verifiable as existing right now); steps (max 6) each with a gradeable done-condition; a Gate command; and a `Numbers basis:` line where every scalar names its basis (measured, derived, or hypothesis with crash-and-halve recovery). Assess the frontier for `/harness` and `/best-of-n` and set one `Harness:` and one `BoN:` field with a one-line rationale each. Use `Harness: yes` for high correctness stakes (security, data integrity, complex multi-file logic, public API contracts, probe/harness methodology). Use `BoN: yes - <rationale> [N=K]` (default N=3, max N=5) when multiple credible designs exist and picking wrong is costly. Use `no` for scaffolding, config, docs, simple CRUD. Assign at most one `yes` across the two fields; a frontier tagged both is malformed. If `Harness: yes`, write a `#### Harness Criteria` subsection; if `BoN: yes`, write a `#### BoN Criteria` subsection with criteria observable from a candidate's git diff alone. Treat each criteria block as a rubric in the managed-agents sense (https://platform.claude.com/docs/en/managed-agents/define-outcomes): explicit, gradeable conditions in the format `C1: [specific observable requirement] -> PASS if [condition], FAIL if [condition]`. Write gradeable observations such as 'CSV contains a price column with numeric values' over vague judgments such as 'data looks good'. When criteria feel hard to author, simulate a known-good implementation and derive criteria from the observable properties that make it good. For each criterion, mark whether it is satisfiable by pure implementation or needs an architectural primitive, as a trailing note on the criterion line. For empirical or experimental frontiers, append a statistical-validity pre-check note inside the criteria block (free-parameter headline, denominator adequacy) to run before any GPU spend, per the Pre-Check Gate in reference/load-bearing-iteration.md. Where the Trail contradicts the Territory sketch, say so explicitly and propose the redraw; the sketch is provisional, the Trail is evidence.

## PDMC pass (mandatory: every `Harness: yes` frontier, before implementing)

Send to codex in the background (`< /dev/null`), with the frontier text and `references/pdmc-checklist.md`:

> Act as a methodological reviewer focused on PDMC methodology. Leave procedural review to `/codex-review --plan`. Apply PDMC items 1-15 from `references/pdmc-checklist.md` to this frontier. Classify each item as PASS, FAIL, or N/A with evidence. Return a PASS/FAIL verdict per item plus an aggregate verdict.

Paste the per-item verdicts and `Aggregate verdict: PASS` into the frontier's `#### PDMC Methodological Review` subsection; any applicable FAIL blocks the frontier until reworked and re-passed.
